# Kreya Codebase Guidelines

## Project Overview

**Kreya** is an AI-powered WhatsApp-first social media management SaaS platform that enables content creators to manage their entire social presence through conversational AI.

**Core Value Proposition:**
- Send a WhatsApp message or voice note describing content needs
- Claude AI generates captions, optimizes format for each platform
- Creators approve/edit via WhatsApp, then publish to Instagram, TikTok, Twitter, Facebook, LinkedIn, YouTube, etc.

**Status:** Early-stage startup with dual tech stacks:
- **Frontend/Marketing**: Static HTML/CSS/JS landing page (GitHub Pages)
- **Application Layer**: Next.js 16+ with React 19, TypeScript, Tailwind CSS

## Code Conventions

### When to Write Code (and When Not To)
**DO:**
- Fix bugs with minimal, targeted changes
- Add new features in isolation
- Use existing patterns as reference

**DON'T:**
- Rewrite entire files for small fixes
- Add unnecessary abstractions or helper functions
- Create utility files for single-use logic
- Add comments explaining WHAT code does (well-named identifiers should suffice)
- Add error handling for impossible conditions
- Add backwards-compatibility shims

### Preferred Code Style
- **Conciseness:** Minimal, targeted code changes only
- **Comments:** Only when the WHY is non-obvious (hidden constraint, workaround for specific bug)
- **Naming:** Clear, self-documenting variable/function names
- **DRY:** Three similar lines is acceptable; premature abstraction is not

### React/Next.js Patterns
- Use React Server Components by default (app router)
- Client Components: Mark with `'use client'` only when needed (interactivity)
- Avoid inline styles; prefer Tailwind utility classes
- TypeScript: Export `type` for TypeScript-only constructs

### HTML/CSS in Static Files
- Use the design system tokens from `kreya-brand-identity.html`
- Maintain consistent spacing (24px, 28px, 32px baseline)
- Use CSS Grid/Flexbox for layout (no float-based layouts)
- Include media queries for responsive design

- Key Flows:
  1. WhatsApp message → Sent to Claude API

### API Integration Points
- `api/publish/instagram.ts` – Instagram publishing logic

- Database: Supabase for user accounts, drafts, schedule, analytics

## File Update Guidelines

### Landing Page (`index.html`)
- **Critical:** Use the color tokens and typography hierarchy
- **Brand Styles:** Reference `kreya-brand-identity.html` for component patterns
- **Structure:** HTML sections with clear semantic structure
- **JS:** Vanilla JavaScript for interactivity (no external dependencies)
- **Responsive:** Test on mobile (mobile-first approach)

### Next.js App
- **Assets:** Place images/icons in `public/` directory
- **Styles:** Use Tailwind utility classes (configured in postcss/tailwind)
- **Components:** Keep them small and reusable; prefer composition
- **Types:** Export interfaces/types in separate files when shared

### Static Artifact HTML Files
- Used for documentation, guides, and component reference
- Format: Self-contained HTML with embedded CSS/JS
- Purpose: Reference material for brand consistency and feature documentation
- Do NOT rely on these for the production application

## Performance & Quality

- ESLint: Run before commits (catches style/type issues)

- [ ] Changes are minimal and targeted
- [ ] TypeScript types are correct
- [ ] No unnecessary imports
- [ ] No console.logs or debug code

- [ ] Linter passes (`npm run lint`)

- No Jest/Vitest setup yet (add when needed)

## Memory Management

Long chat sessions may hit context limits. When this happens:
1. Summarize progress in a brief note
2. Ask user to start a new Claude session
3. Reference previous work and continue from there

This keeps each session focused and context efficient.

## Project Structure Philosophy

**Dual Tech Stack (Intentional):**
- **Marketing/Public Site:** Static HTML (fast, no build, GitHub Pages friendly)
- **Application:** Next.js (full-stack, real-time, user accounts, API)

This keeps the public landing page decoupled from the app, enabling:
- Fast static landing page updates
- Complex app features without compromising marketing site performance
- Clear separation of concerns

1. Edit color tokens in `index.html` `:root`
2. Update font sizes/weights in relevant sections
3. Test responsive design (680px breakpoint)

**Add a new API endpoint:**
```
api/route-name/handler.ts     # Create endpoint
// Export POST/GET/etc. handlers
```

- Edit inline Tailwind classes (no custom CSS needed)
- Verify with `npm run dev` locally
- Check responsiveness at key breakpoints
