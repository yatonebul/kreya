Step 1 — Create your Meta App

This is the one-time setup on Meta's developer portal. Takes about 15 minutes. The API itself is free — Meta does not charge for Graph API access.

⚡
Before you start: Your users need Instagram Business or Creator accounts. Personal accounts cannot connect. Add this requirement clearly in your Kreya onboarding.

1.1 Create a Meta Developer account

Go to developers.facebook.com → Log in with your Facebook account → Register as a developer (if not already).

1.2 Create the app

  • Click Create App → choose Business as app type
  • App name: Kreya (or anything — users won't see this during OAuth)
  • Contact email: your email
  • Click Create App → complete security check

1.3 Add Instagram Platform product

  • In your app dashboard → Add Product
  • Find Instagram Platform → click Set Up
  • Choose "Instagram API with Instagram Login" (not Graph API — that's the old one requiring a Facebook Page)
✓
Instagram Platform API (July 2024) — this is the new direct login API. Users authenticate with Instagram directly, no Facebook Page required. Works for Business and Creator accounts.

1.4 Configure OAuth redirect URIs

In Instagram → API Setup with Instagram Login → OAuth Redirect URIs, add both:

Redirect URIs to add in Meta Console
http://localhost:3000/api/instagram/callback
https://kreya.ai/api/instagram/callback

1.5 Set required permissions

In Instagram → Permissions and Features, request:

  • instagram_business_basic — read profile info
  • instagram_business_content_publish — post content
ℹ
During development, you can test with your own IG account as a test user — no App Review needed. You only need to submit for App Review when you want real users to connect. Review takes 1–4 weeks but runs in the background while you build.

1.6 Get your credentials

Go to Instagram → API Setup with Instagram Login. Copy two values:

  • Instagram App ID — paste as INSTAGRAM_APP_ID
  • Instagram App Secret — paste as INSTAGRAM_APP_SECRET
⚠
The Instagram App ID is different from the Facebook App ID shown in the main app settings. Make sure you copy from the Instagram product section.

2.3 What each file does

lib/instagram/crypto.ts
Token Encryption
AES-256-CBC. Encrypts tokens before DB. Decrypts server-side only before API calls. Plaintext never touches the database.
lib/instagram/oauth.ts
OAuth + Token Management
Builds auth URLs, exchanges codes, gets long-lived tokens (60 days), handles refresh logic.
lib/instagram/publish.ts
Publishing Engine
2-step publish (container → publish). Handles images, videos, Reels, carousels. Polls until container ready.
api/instagram/oauth
OAuth Start
Generates secure state, stores in Redis, redirects user to Instagram.
api/instagram/callback
OAuth Return
Validates state (CSRF), exchanges code, encrypts token, saves to Supabase.
api/instagram/post
Publish Content
Auth check, plan limits, rate limit, decrypt token, publish, save record, audit log.

Step 3 — Run the database migration

Open your Supabase Dashboard → SQL Editor → New query → paste and run:

supabase/migrations/003_instagram_direct.sqlSQL
CREATE TABLE instagram_accounts (
  id                    UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  org_id                UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  instagram_user_id     TEXT NOT NULL,
  username              TEXT NOT NULL,
  name                  TEXT,
  profile_picture_url   TEXT,
  followers_count       INT DEFAULT 0,
  account_type          TEXT CHECK (account_type IN ('BUSINESS','CREATOR','PERSONAL')),
  is_active             BOOLEAN DEFAULT TRUE,
  encrypted_token       TEXT NOT NULL,
  token_iv              TEXT NOT NULL,
  token_expires_at      TIMESTAMPTZ NOT NULL,
  connected_at          TIMESTAMPTZ DEFAULT NOW(),
  disconnected_at       TIMESTAMPTZ,
  UNIQUE (org_id, instagram_user_id)
);

CREATE TABLE instagram_posts (
  id                    UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  org_id                UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  instagram_account_id  UUID NOT NULL REFERENCES instagram_accounts(id),
  caption               TEXT NOT NULL,
  media_type            TEXT NOT NULL CHECK (media_type IN ('IMAGE','VIDEO','REEL','CAROUSEL')),
  media_urls            TEXT[] NOT NULL,
  status                TEXT NOT NULL DEFAULT 'pending'
                        CHECK (status IN ('pending','published','failed','scheduled')),
  instagram_media_id    TEXT,
  instagram_permalink   TEXT,
  scheduled_for         TIMESTAMPTZ,
  published_at          TIMESTAMPTZ,
  error_message         TEXT,
  retry_count           INT DEFAULT 0,
  created_at            TIMESTAMPTZ DEFAULT NOW()
);

ALTER TABLE organizations
  ADD COLUMN IF NOT EXISTS monthly_ai_posts_used INT DEFAULT 0,
  ADD COLUMN IF NOT EXISTS monthly_reset_at TIMESTAMPTZ DEFAULT NOW();

CREATE INDEX idx_ig_accounts_expires ON instagram_accounts (token_expires_at) WHERE is_active = TRUE;
CREATE INDEX idx_ig_accounts_active  ON instagram_accounts (org_id, is_active);
CREATE INDEX idx_ig_posts_org        ON instagram_posts (org_id, created_at DESC);

ALTER TABLE instagram_accounts ENABLE ROW LEVEL SECURITY;
ALTER TABLE instagram_posts    ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users see own IG accounts" ON instagram_accounts FOR SELECT
  USING (org_id = auth.uid());
CREATE POLICY "Users see own IG posts" ON instagram_posts FOR SELECT
  USING (org_id = auth.uid());
✓
RLS policies ensure users can only ever read their own data. Even if a bug exposed the raw query, User A cannot see User B's connected accounts or posts. Insert/update is done server-side using service role key which bypasses RLS intentionally.

# Token encryption — 32+ chars, generate with:
# openssl rand -base64 32
TOKEN_ENCRYPTION_KEY=your-32-plus-char-random-string

# Upstash Redis (upstash.com → Create Database → REST API)
UPSTASH_REDIS_REST_URL=https://your-db.upstash.io
UPSTASH_REDIS_REST_TOKEN=your-upstash-token

# Upstash QStash (for nightly token refresh cron)
QSTASH_CURRENT_SIGNING_KEY=sig_xxx
QSTASH_NEXT_SIGNING_KEY=sig_xxx
QSTASH_TOKEN=your-qstash-token

Generate the encryption key

terminalShell
openssl rand -base64 32
# Example output: dK7bX9mN3pQ8rV2sL5wY1cA6eH4jF0gT
# Copy this output as TOKEN_ENCRYPTION_KEY
⚠
Critical: Never change TOKEN_ENCRYPTION_KEY after users have connected accounts. Changing it makes all stored tokens undecryptable — every user would need to reconnect Instagram.

Step 5 — Set up nightly token refresh

Instagram long-lived tokens expire after 60 days. The refresh route runs nightly via Upstash QStash — a serverless cron service. Free tier covers up to 500 messages/day which is more than enough.

5.1 Create the QStash schedule

upstash.com → QStash → Schedules → Create
URL:      https://kreya.ai/api/instagram/refresh
Method:   POST
Schedule: 0 3 * * *     # Every day at 3am UTC
Headers:  none needed   # QStash adds its own signature header automatically

5.2 Or via curl (one-time setup)

terminalShell
curl -X POST https://qstash.upstash.io/v2/schedules \
  -H 'Authorization: Bearer YOUR_QSTASH_TOKEN' \
  -H 'Content-Type: application/json' \
  -d '{"destination":"https://kreya.ai/api/instagram/refresh","cron":"0 3 * * *"}'
✓
The refresh route is protected by QStash signature verification — verifySignatureAppRouter checks the cryptographic signature on every request. Nobody else can trigger it, even if they know the URL.

5.3 Reset monthly post counters

Add a second QStash schedule to reset monthly usage on the 1st of each month:

QStash second schedule
URL:      https://kreya.ai/api/billing/reset-monthly
Method:   POST
Schedule: 0 0 1 * *     # 1st of every month at midnight UTC

6.4 Common errors and fixes

Error reference
Error: "state_mismatch"
→ Redis not configured or state expired (>10min). Check UPSTASH_REDIS_REST_URL.

Error: "token exchange failed"
→ INSTAGRAM_APP_ID or SECRET wrong. Check Meta Console → Instagram section (not Facebook section).

Error: "URL must be accessible" / error code 36003
→ mediaUrl is not publicly accessible. Upload to Supabase Storage public bucket first.

Error code 190: "Invalid OAuth access token"
→ Token expired. User needs to reconnect. Frontend should redirect to /api/instagram/oauth.

Error: "This endpoint requires the 'instagram_business_content_publish' permission"
→ Permission not approved yet. Add yourself as test user in Meta Console → Roles.

Error: "redirect_uri does not match"
→ INSTAGRAM_REDIRECT_URI in .env.local must exactly match what's in Meta Console.
   Trailing slashes, http vs https, port numbers — must be identical.
